# Portfolio Expense Visualizer

## 概要
銀行・クレジットカードの明細CSVを取り込むことで、
月ごとに「何に」「いくら」支出が発生しているのかを集計し、
カテゴリ別・月次で可視化する個人向け家計簿管理アプリケーションです。

## 使用技術

- Java / Spring Boot  
  業務アプリケーションで広く採用されており、MVC構成およびレイヤード構成を意識した設計・実装が可能なため採用した。

- PostgreSQL  
  集計や日付処理を含むSQLを安定して扱えるRDBであり、本アプリケーションの月次・カテゴリ別集計という要件を過不足なく満たすため採用した。  
  また、OSSで導入障壁が低い点も理由の一つである。

- MyBatis  
  表示対象となるデータの抽出条件や結合条件をSQLで定義しており、アプリケーションの振る舞いそのものをSQLが担っているため、SQLを明示的に記述できるMyBatisを採用した。

## 機能一覧

- 銀行・クレジットカードの明細を取り込み、支出データを登録できる
- 支出カテゴリを追加・編集・削除できる
- 支出データをカテゴリ別に管理できる
- カテゴリ別の支出割合を円グラフで確認できる
- カテゴリ別の支出金額を月ごとに棒グラフで確認できる
- 月次支出の推移を折れ線グラフで確認できる
- 当月・前月の総支出額および前月比を確認できる

## 工夫した点

- CSV取り込みでは、銀行やクレジットカードごとに明細フォーマットが異なることを前提とし、将来的な変更にも対応しやすい柔軟な構成を意識してライブラリ選定および実装を行った。

- サマリー（ダッシュボード）画面では、表示に必要な集計やデータ加工処理をJava側に集約し、テンプレート側は値を表示するだけの役割にすることで、責務の分離を意識した設計とした。

- グラフ表示については、データ取得はサーバー側、描画はフロントエンド側と役割を分け、Chart.jsの責務に委ねる構成とした。また、カテゴリ管理ではすべてのデータにCRUD機能を持たせるのではなく、利用シーンに応じて必要な操作のみに限定することで、画面や操作が複雑にならないよう配慮した。

## 苦労した点・学んだこと

- データベース設計やSQLの記述では、テーブル間の関係や制約を意識する必要があり、単にデータを取得するだけでなく「どのような構造で保持すべきか」を考える点に苦労した。実装を通じて、集計処理や将来的な拡張を見据えたテーブル設計の重要性を学んだ。

- アプリケーションのデプロイでは、ローカル環境で動作していたものを本番環境で動かすために、環境変数やポート設定、データベース接続など多くの設定が必要であることを理解するまでに時間を要した。調査と試行錯誤を通じて、開発と運用は切り離せない工程であることを学んだ。

## 今後の改善点

- 実装を進める中で「本来はこうしたかった」という修正が発生し、手戻りが生じる場面があったため、着手前の設計が不十分だったと感じている。  
  今後は実装に入る前に、画面遷移や機能一覧を整理し、どの画面で何を行うかを明確にした上で開発を進めたい。

- また、機能追加の判断が実装中に発生することが多かったため、最初に「今回実装する範囲」と「実装しない範囲」を明確に定義することで、開発中の迷いや手戻りを減らしたい。
